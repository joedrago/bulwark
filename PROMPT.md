Then perhaps we can move onto reading a settings toml file for each executable, and make sure that data is available correctly, etc. We can work our way into a basic game UI state machine where we can see a splash screen, a basic main menu, menu selections, basic inputs from keyboard/mouse/gamepad, etc. "top-down" means we verify the really important plumbing and "engine" before we ever get to the actual gameplay and game state.

* "with history" means I plan to manually verify as a human each phase and do git commits along the way. The idea here is that I want to have a long historical record in version control of every decision we made, and be able to go back and see where we made mistakes. I can't do this if Claude just "one-shots" everything and it is a big pile of soup I don't understand.


# Language / Library / Build Output

I want the two programs that this project will ultimately make to be written entirely in Rust, formatted with the standard Rust formatter. Ideally, this project will build cleanly on the 3 major OSes (Windows, macOS, Linux) using just a clone of this repo and a standard Rust installation. The build should have a means to create a "package" for the current OS which is functionally "portable". As an example, a Windows build might be 2 .exe files and a handful of .toml files, which all sit in a single directory and can be delivered to people as such. It never needs any data outside of this directory. If we decide to not embed art/sound/music, there can be a data/ directory next to them that is used, of course. There shouldn't be any usage of awful OS constructions like APPDATA, or registries, etc. The only caveat here would be that the macOS build can be entirely packaged into a .app directory structure containting the data and toml settings files, etc, and if the user preferences toml file (graphics/sound settings, etc) must be stored in macOS's appdata equiv, that might OK. we can tackle that decision later.

Ideally these builds don't demand any special tech on the OS side, so I'd prefer to use Rust libraries that try to use built-in OS stuff as much as possible. This will be an entirely 2D game, but for performance it probably makes sense to use some kind of cross platform basic graphics acceleration like OpenGL or SDL or something. We will need to be able to let the user configure their graphics mode (windowed, fullscreen, windowed fullscreen, resolution, refresh rate), keybinds, gamepad bindings, etc. We will need basic networking for sending game state across a network boundary in an efficient and consistent manner, but nothing too overarchitected. We're not talking about a ton of data, just a big grid that we can probably pack pretty efficiently, some animation state, some states, some coordinates, etc. We can figure that out more precisely later too.

# Glossary

In order for us to be able to efficiently communicate about the future architecture of this game, we will need a shared vocabulary. When using these terms with a capital letter, we should both assume the other is talking about the term defined here. Let's agree on these terms now:

* App - This is the term used to refer to the executable that lets the player play the game. It provides the graphics and sound and input for the enduser, and is the main window into all usage of this project.

* Dedicated - This is the term used to refer to the executable that runs on a remote server and allows many separate groups of players to play simultaneously. This is not a noun but it is being used as such, as a shorthand reference to "dedicated server". I am not calling it "server" though as I plan to use that term as an internal piece of architecture and I want a clear distinction between them when we talk.

* Client / Server / Room / Owner - I am defining these terms simultaneously because they only make sense simultaneously. Despite this game being playable entirely locally against computer players, architecturally there is an important wall between the notion of a "Client" side and a "Server" side, much like how Quake 3 built its engine. When the Client wants to play a game, it contacts the appropriate Server (either embedded in its App or over the network to a Dedicated, depending on UI choices) and requests a new Room. that Room inside of the Server is the source of truth for all game state, animations, who is winning, losing, score, what phase the game is in, etc. The Client which requests this Room becomes the Owner of the Room inside of the Server. Of course, for an entirely local situation where only the App is involved and no networking is needed, this will feel like a bit of overcomplication, but it is important that we don't have any significant special cases so that real network multiplayer is rock solid and bugfree. The Server and all Clients are responsible for fast an efficient communication with each other to ensure that the Client has an appropriate readonly copy of the game state, and the Clients will basically perform actions with their inputs which will be sent to the Server's Room code, which may or many not update the game state (or signal an error to that specific client, etc), and then that Room's new state must be broadcast appropriately to all Clients so every Client sees the same state and can make decisions in realtime. When determining high level things like starting a new game or quitting, the Owner will have special privileges around the Room it created.

We should be able to have a phase where we run the App fresh, request a "New Game (Local)", and this will cause the Client to talk to its internal (no networking) embedded Server which will grant it a local Room. It then uses the communication between Client and Server to manipulate the Room state and underlying gamestate until the user "leaves" the room, in which the embedded Server will simply abandon that Room. An App's internal Server will only ever have zero or one Room states active, but architecturally shouldn't consider this a special case.

We should then be able to run the App fresh, request a "New Game (Multiplayer)", and instead of using some internal non-network connection between Client and the App's embedded Server, it will initiate a network connection to the preconfigured Server running inside of a Dedicated at some network host. Upon connecting, it will request a new Room, which the Dedicated will simply manage alongside its other rooms, and make it clear what the 4 random letter name of the Room is. Other instances of the App (likely running on other player's machines, but possibly locally on a developer's machine for debugging) can choose "Join Game" and type in the random 4 letter name of the Room (communicated external to the game by human players), and then connect to that Server in the Dedicated and join that Room as well. The first Client will be somehow known to be the Owner and will be able to manage the Room appropriately. Meanwhile an entirely different set of Apps running on other machines should independently and simultaneously be able to connect to the single Dedicated and request their own Room, which the Dedicated will also manage.

I would like the Client/Server architecture to be lean and mean but resilient. The Room and game state isn't that complicated -- we should be able to tolerate Clients being temporarily being disconnected from the Server and Room, and waiting up to some reasonable timeout before abandoning the Room. Upon reconnecting, it should simply remind the Client of all Room and game state and the game can continue. This might require that each "App" has its own unique ID somehow, so perhaps when you launch the App, if it doesn't see a unique ID string (some long SHA looking string) inside of its user settings, it generates one and stores it in there. We can have a debug-time commandline argument for the App executable to override it so we can run multiple App on a single machine for debugging. It is reasonable to think actual endusers are only ever going to have one instance of App running on their machine.

* Player - This is simply the noun we use for a human being playing the game.

* Bot - a Bot is a "computer Player". They are a Server/Room notion, and spawned via an Owner request. If during the setup of a new Room (the lobby state, effectively) the Owner requests that a computer player be added to the game, the Server should add a Bot to the Room, which has its own random name, perhaps eventually a difficulty selection by the Owner, etc. Once it is there though, it should appear to just be another Player in the Room. When gameplay actually happens, the Server and Room code will think for the Bot and deliver the same kinds of actions that a real player would. This might be difficult to implement and can be a much later phase, as long as we don't make tech decisions that block such a thing existing.

* Animation - We can generally use the lowercase "animation" when referring to any kind of animation in the game and UI, but "Animation" with a capital A should refer to a specific kind of Room state which allows for synchronized animations to play on all Clients. They can be quite specific with a big enum describing them (FIRE, CANNONBALL, etc), and have a notion of a start time (in some kind of Client/Server agreed mono time in milliseconds), duration, various coordinates necessary for interpolation, etc. The idea here is if the Server/Room decides that a cannonball is now in flight between coordinate (3,5) and coordinate (18, 23) and it should take 6 seconds to fly, that can be added in the Animation list in the Room with some kind of ever-incrementing id, and all Clients can check against their local Animation entries to make sure they're tracking and drawing that, and if not, update their Client render to show it. Any other tech like this can be similarly added (sound effects, etc). It can't be ever-growing though, of course. The Server/Room tech must know how to clean these up and age them out when they aren't used anymore, and be cognizant of animations still in flight (especially running on remote Clients) before moving onto the next phase, etc, or perhaps not updating grid state with a blown up wall until the associated Animation of the cannonball in flight finishes, etc. Be creative with this tech, but keep it simple.


# The Game

As a general rule, once Players and possibly Bots are added to a Room and the game is started by the Owner, the game should play almost identically to the original arcade game Rampart. There should be a notion of nearby islands, all visible to all players on a single zoomed out screen, separated by water, and effectively on a large shared grid. There should be some kind of fanfare signaling that a build phase is incoming (music, some kind of animation, etc), then players have a specific amount of seconds to place randomly generated (but identical in order to all Players / Bots for fairness) pieces in their own island's grid. If they successfully surround one of the castles on their island, it will flood fill the ground signaling that they have now have that area claimed. When the timer for the build phase runs out, anyone that hasn't surrounded one castle is out of the game ("Game Over") and must simply watch the other players continue playing. If only one person is left, they win the game. If more than one person is left, they move onto the cannon placement phase, where the quantity of cannons they can place is dictated by how much area they currently claim. They place cannons, then there's a battle phase where players can shoot their cannons (each fire request shoots one cannonball from an available cannon that doesn't already have a cannonball in flight) at their opponents walls and cannons, trying to destroy them. When that battle phase is done, another floodfill style area check occurs and we go back to the build phase.

**When in doubt, make Rampart.** - This includes music cues, sound cues, UI animation, cursors, etc. As this is a phased project, we can approach each piece as we go and interate for fun.

The biggest departure from Rampart is the quantity of players allowed. I would like to allow up to 6 players, if possible. Rampart generally bisects a single large grid into two equal diagonal halves for 2 players, and then more of a pie split for 3 players, where each player still has a reasonable grid to work with. I would like to continue this pattern up to 6 players, where the grid gets larger (and simply zooms out to accomodate the bigger world), and we just cut like they were pie slices, preferably with a bit of wiggle room so it looks natural.

Other than the increased player count and the zoomed grid/visuals that need to accomodate that, Bulwark is just Rampart.


# The Input

I want Keyboard players, Mouse players, and Gamepad players to all be first class citizens in this simple game. All menus should be navigable entirely with the gamepad and keyboard, and all keybindings and buttons should be configurable with sensible defaults. Generally speaking we're really only going to need LRUD, Accept(/Fire) and Cancel, so it shouldn't be tough. Mouse players will probably just use left and right click for Accept and Cancel, respectively, and/or perhaps let space bar rotate for people playing on a single button mouse.

Gamepad and keyboard players will operate strongly on a grid, where one tap of dpadleft or left arrow (whatever binding they choose) will move the cursor in-game on grid unit, for precision. Mouse players will track the cursor and simply try to choose the closest place under the cursor. During the battle phase, gamepad analog sticks can have the opportunity to behave more like a mouse cursor and be a bit more fluid, but we should probably make this configurable and start with it simply being interpreted as LRUD.

Any UI drawn to indicate user input (a ghost piece floating over the grid, a firing cursor, etc) is entirely a Client driven thing, but should also regularly be communicated to the Server/Room so that other Clients have access to the rough position and piece of other players. Their Clients should be able to show what their opponents are doing and aiming at.
